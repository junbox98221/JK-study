# 11. Reconciler의 스케줄링 사전 작업

> 이번 챕터에선 Reconciler가 Scheduler에 Work를 넘기기 전에 수행하는 사전 정리 작업을 코드를 통해 살펴봅니다.

## 1. Reconciler와 Scheduler의 역할

React에서 Work의 실행 제어권은 Scheduler가 제어합니다.
Reconciler는 **필요한 정보를 정리해 Scheduler에게 넘기는 역할**을 수행하고, 실제 실행 시점은 Scheduler가 결정합니다.

- **Reconciler**: 스케줄링 정보를 수집하고 기록합니다.
- **Scheduler**: 브라우저 상태와 우선순위를 고려해 실행 시점과 중단 여부를 결정합니다.

Scheduler 내부에는 **host config**라는 호스트 환경 의존 모듈이 있으며, `performance`, `isInputPending` 같은 브라우저 API를 다룹니다.
특히 `isInputPending`을 통해 **정말 양보가 필요한 순간에만 중단**할 수 있어, Concurrent Mode에서 불필요한 제어권 양보를 줄일 수 있습니다.

## 2. `dispatchAction`에서 `scheduleWork`로 이어지는 흐름

Hooks에서 상태 업데이트가 발생하면 `dispatchAction` → `scheduleWork` 순으로 호출됩니다.

```javascript
// /react/packages/react-reconciler/src/ReactFiberHooks.js

function dispatchAction(...) {
  if (...) {
    /* Render phase update... */
  } else {
    /* idle update... */
    scheduleWork(fiber, expirationTime);
  }
}
```

`scheduleWork`는 `scheduleUpdateOnFiber`에 연결되어 있으며, 가장 먼저 `markUpdateTimeFromFiberToRoot`를 호출합니다.

```javascript
// /react/packages/react-reconciler/src/ReactFiberWorkLoop.js

export function scheduleUpdateOnFiber(fiber, expirationTime) {
  const root = markUpdateTimeFromFiberToRoot(fiber, expirationTime)
}

export const scheduleWork = scheduleUpdateOnFiber
```

Reconciler는 Scheduler에 넘기기 전에 다음 세 가지를 수행합니다.

1. **업데이트 발생 시점을 나타내는 `expirationTime`을 기록합니다.**
2. **이벤트가 발생한 Fiber로부터 VDOM의 root를 찾습니다.**
3. **해당 root에 스케줄링 판단에 필요한 정보를 반영합니다.**

## 3. `markUpdateTimeFromFiberToRoot`가 하는 일

이 함수는 **업데이트가 발생한 Fiber부터 root까지 시간 정보를 마킹**합니다.

```javascript
// /react/packages/react-reconciler/src/ReactFiberWorkLoop.js

function markUpdateTimeFromFiberToRoot(fiber, expirationTime) {
  // expirationTime을 새긴다.
  if (fiber.expirationTime < expirationTime) {
    fiber.expirationTime = expirationTime
  }
  let alternate = fiber.alternate
  if (alternate !== null && alternate.expirationTime < expirationTime) {
    alternate.expirationTime = expirationTime
  }

  // root를 찾는다.
  let node = fiber.return
  let root = null
  if (node === null && fiber.tag === HostRoot) {
    root = fiber.stateNode
  } else {
    while (node !== null) {
      alternate = node.alternate
      // childExpirationTime을 새긴다.
      if (node.childExpirationTime < expirationTime) {
        node.childExpirationTime = expirationTime
        if (
          alternate !== null &&
          alternate.childExpirationTime < expirationTime
        ) {
          alternate.childExpirationTime = expirationTime
        }
      } else if (
        alternate !== null &&
        alternate.childExpirationTime < expirationTime
      ) {
        alternate.childExpirationTime = expirationTime
      }

      if (node.return === null && node.tag === HostRoot) {
        root = node.stateNode
        break
      }
      node = node.return
    }
  }

  return root
}
```

### 3-1. Fiber에 `expirationTime` 기록

업데이트가 시작된 Fiber와 `alternate` 양쪽에 `expirationTime`을 새깁니다.
React는 **current / workInProgress** 두 트리를 관리하므로, 한쪽만 갱신하면 정보가 어긋날 수 있어 둘 다 처리합니다.

`expirationTime`은 해당 업데이트가 **얼마나 시급하게 다뤄져야 하는지**를 나타내는 기준값입니다.
이후 lane 모델로 발전했지만, 업데이트에 우선순위 기준이 필요하다는 핵심 개념은 같습니다.

### 3-2. 부모 방향으로 `childExpirationTime` 기록

부모를 타고 올라가면서 각 조상 노드에 `childExpirationTime`을 기록합니다.
이 값은 **"이 노드의 자손 어딘가에 해당 시점의 업데이트가 존재한다"** 는 뜻으로, 재조정 시 어느 서브트리를 우선 탐색해야 하는지 판단하는 데 쓰입니다.

### 3-3. root를 찾아 반환

최종적으로 `HostRoot`에 도달하면 `stateNode`(= root)를 반환합니다.
여러 업데이트가 동시에 발생할 수 있으므로, 스케줄링 조율의 기준점이 되는 **root에 정보를 모아두는 것**이 핵심입니다.

## 4. 정리

1. `dispatchAction` → `scheduleWork`(`scheduleUpdateOnFiber`) → `markUpdateTimeFromFiberToRoot` 순으로 호출됩니다.
2. 업데이트가 발생한 Fiber와 alternate에 `expirationTime`을 기록합니다.
3. 부모로 올라가며 각 조상에 `childExpirationTime`을 기록합니다.
4. 최상위 `HostRoot`를 찾아 root를 반환합니다.
5. 이 정보를 바탕으로 Scheduler가 우선순위와 실행 시점을 판단합니다.
