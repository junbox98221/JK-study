# 10. 훅 상태 업데이트 요약 및 스케줄러 개관

> 이번 챕터에선 지난 챕터들에서 학습한 훅(Hook)에 대해 간단히 되짚고, 앞으로 학습할 스케줄러(Scheduler)의 역할을 개관합니다.

## 1. 훅(Hook)을 통한 상태 업데이트 과정 복습

지난 챕터에서 훅을 통해 컴포넌트 상태를 업데이트하는 과정을 상세히 살펴보았습니다.

### 1.1. 훅 구현체 탐색 및 생성 과정

- `shared` 및 `reconciler` 패키지를 따라가며 훅 구현체 확인
- 컴포넌트가 마운트될 때 훅이 생성되는 과정

### 1.2. 상태 변경 및 리렌더링 과정

- `setState` 함수 호출을 통해 상태를 업데이트하고 컴포넌트를 리렌더링하는 과정
- `setState` 함수는 idle 상태와 render 상황에서 다르게 동작
- idle 상태에서 `setState` 호출의 마지막은 업데이트 적용을 위한 `scheduleWork` 함수 호출로 이어짐

## 2. 스케줄러(Scheduler)와 리컨실러(Reconciler)의 역할 개관

훅으로 상태가 바뀌면, 그 변경을 실제 화면에 반영하는 과정은 리컨실러와 스케줄러가 나누어 담당합니다.

- 상태 업데이트가 일어나면, 마지막에 `scheduleWork`가 호출되며 업데이트 작업이 시작됩니다.
- **리컨실러(Reconciler)** 는 어떤 작업을 해야 하는지 정리하고, 실제 실행 시점은 스케줄러에 넘깁니다.
- **스케줄러(Scheduler)** 는 브라우저 상태와 우선순위를 고려해, 작업을 언제 실행할지 결정합니다.
- 이때 스케줄러가 환경에 맞게 동작할 수 있도록 **호스트 컨피그(Host Config)** 가 필요한데, 여기에는 브라우저나 모바일 같은 각 실행 환경에 맞는 API와 설정이 담겨 있습니다.
